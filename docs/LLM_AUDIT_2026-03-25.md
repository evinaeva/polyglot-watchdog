# LLM Integration Audit (2026-03-25)

## Historical context (important)

This audit captured repository findings **before** the follow-up telemetry/UI remediation landed.  
It is preserved as historical evidence and **must not** be read as the current repository state.

Post-remediation note:
- The duplicate `/check-languages` telemetry render path was removed in favor of one canonical LLM review block.
- The `tests/test_check_languages_page.py` collection issue (duplicate test function declaration) was fixed.
- A gated real-provider validation test was added for Phase 6 (`tests/test_phase6_llm_integration_real_provider.py`), but live provider success remains environment-dependent.
- Operator-facing hints are now surfaced when `review_mode=llm` but no real provider request was executed, and `operator_notes` aggregate fallback/response conditions.

This document records a strict code-and-runtime audit of:

1. Backend LLM telemetry + batching + token/cost accounting.
2. UI rendering of LLM process and statistics on `/check-languages`.

## Verdict

**Not ready for production LLM-based QA.**

Core blockers:
- Real LLM execution cannot be proven in this environment because `PHASE6_REVIEW_PROVIDER` and `PHASE6_REVIEW_API_KEY` are not configured.
- The `/check-languages` UI has two different telemetry render paths with incompatible schemas; one expects legacy keys and can show malformed warnings even when backend telemetry is valid.
- Tests around check-languages telemetry are currently not runnable due a syntax error in `tests/test_check_languages_page.py` (duplicate function declaration).

## Evidence Summary

- Phase 6 requires explicit review mode via `--review-mode` or `PHASE6_REVIEW_PROVIDER`; missing value raises a hard error.
- In async check-languages orchestration, missing `PHASE6_REVIEW_PROVIDER` fails the job in `running_comparison_failed` stage.
- LLM provider uses batching with an estimated safe context budget and fallback behavior on transport/provider/parse failures.
- `llm_review_stats.json` is always written at end of `run_phase6.run` from `provider.get_llm_review_stats()` (or empty dict with `review_mode` patched).
- UI has both:
  - modern rendered block (`_llm_review_display`) that understands current telemetry shape, and
  - legacy strict block (`llm_review_stats_block`) that expects old keys (`provider`, `model`, `prompt_tokens`, etc.).

## Runtime Attempt (Real Execution)

Attempted to validate live execution by checking required environment:
- `PHASE6_REVIEW_PROVIDER`: missing
- `PHASE6_REVIEW_API_KEY`: missing

Result: **BLOCKED**. No production provider request could be executed through the real code path.

## Key Risks

1. **Silent non-LLM path in "llm" mode when API key missing**:
   - Batch status becomes `not_attempted`, fallback executes, and top-level `llm_requested` remains false because attempted batches exclude `not_attempted`.
   - This is visible in telemetry but can still be operationally confusing if operators only see "llm" mode set.

2. **UI schema mismatch risk**:
   - backend emits `configured_provider/configured_model/effective_model/estimated_prompt_tokens/...`
   - legacy block expects `provider/model/prompt_tokens/...` and can mark valid telemetry malformed.

3. **Test suite reliability risk**:
   - Syntax error currently prevents collection of `tests/test_check_languages_page.py`.

## Minimal Fixes

1. Keep only one telemetry renderer in `/check-languages` and normalize to current backend schema.
2. Add explicit UI warning when `review_mode=llm` but `llm_requested=false` due missing API key (`provider failure: missing_api_key`).
3. Repair syntax error in `tests/test_check_languages_page.py` and add assertions against the modern telemetry shape.
4. Add one integration test that runs `run_phase6.run(..., review_mode='llm')` with mocked request usage payload and verifies full `llm_review_stats.json` consistency.
